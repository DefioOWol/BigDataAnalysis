# Использование Apache Kafka

Работа основывается на домене и генерации из [ЛР 4](../Lab-4/scripts/).


## Часть 1. Приложение с Kafka

Реализована линейная цепочка из трех топиков:

| Этап | Топик | Смысл |
|------|--------|--------|
| Исходные события | `prices.raw` | Строки `prices.csv` как *JSON* |
| Преобразование | `prices.transformed` | Нормализация: дата `yyyy-MM-dd`, числовая цена, каноническая категория, флаги доступности/промо - по тем же правилам, что и очистка в [ЛР 4](../Lab-4/README.md) |
| Фильтрация | `prices.filtered` | Остаются только записи с валидной датой, ценой \> 0, известной категорией и непустыми `store` / `product` / `product_id` |
| Агрегирование | [файл](./data/kafka/aggregates_by_category.json) (пример результатов) | По сообщениям из `prices.filtered` считаются `rows`, `sum_price`, `avg_price` по категориям; результат в *JSON* |

После потока данных продюсер отправляет служебное сообщение `{"_control": "end"}`: оно проходит
через трансформер и фильтр и служит сигналом агрегатору сбросить накопленную статистику в файл.


### Кросс-платформенность

- **Язык:** Python 3.12, зависимость `kafka-python` ([requirements-kafka.txt](requirements-kafka.txt));
- **ОС:** скрипты не используют специфичные для Windows API; пути через `pathlib`;
- **Способы запуска:** Docker Compose или локально (подробнее ниже).


### Сборка и развертывание (Docker)

1. Перейти в каталог:

   ```bash
   cd Lab-6
   ```

2. Запустить Kafka и дождаться готовности:

   ```bash
   docker compose up -d kafka
   docker compose ps
   ```

3. Создать топики (однократно после старта кластера или после `docker compose down -v`):

   ```bash
   docker compose run --rm kafka_init
   ```

4. Сгенерировать сырые данные (*CSV* / *JSON* в [data/raw/](./data/raw/)):

   ```bash
   docker compose run --rm data_gen
   ```

5. Запустить воркеры (три отдельных процесса - трансформация, фильтрация, агрегация):

   ```bash
   docker compose up -d kafka_transform kafka_filter kafka_aggregate
   ```

6. Запустить продюсер для отправки цен в Kafka:

   ```bash
   docker compose run --rm kafka_producer
   ```

   Важно следить за количеством генерируемых строк, так как это напрямую влияет на время обработки.

7. Результат агрегации: [data/kafka/aggregates_by_category.json](./data/kafka/aggregates_by_category.json);

8. Остановка воркеров и брокера:

   ```bash
   docker compose stop kafka_transform kafka_filter kafka_aggregate
   docker compose stop kafka
   ```

Чтобы полностью сбросить состояние брокера необходимо удалить том:

```bash
docker compose down -v
```


### Локальное использование (без Docker)

1. Установить Python 3.12+, выполнить `pip install -r requirements-kafka.txt` из каталога `Lab-6`;
2. Запустить Kafka с listener (например на `localhost:9092`);
3. Создать топики `prices.raw`, `prices.transformed`, `prices.filtered`;
4. Сгенерировать данные из каталога `Lab-6`: `python scripts/generate_data.py`;
5. В отдельных терминалах из каталога `Lab-6`:

   ```bash
   cd Lab-6
   python scripts/kafka_transform.py
   python scripts/kafka_filter.py
   python scripts/kafka_aggregate.py
   python scripts/kafka_producer.py
   ```

   Порядок: сначала воркеры, затем продюсер.


### Назначение файлов

- [compose.yaml](./compose.yaml) - Kafka, инициализация топиков, воркеры, генератор, продюсер;
- [domain.py](./scripts/domain.py) - константы домена и имена топиков;
- [generate_data.py](./scripts/generate_data.py) - генерация данных;
- [kafka_common.py](./scripts/kafka_common.py) - общий код Kafka;
- [kafka_producer.py](./scripts/kafka_producer.py), [kafka_transform.py](./scripts/kafka_transform.py), [kafka_filter.py](./scripts/kafka_filter.py), [kafka_aggregate.py](./scripts/kafka_aggregate.py) - этапы пайплайна.

## Часть 2. Бизнес-план перехода к инфраструктуре больших данных

**Компания:** Региональная сеть гипермаркетов (500 магазинов), источники данных:
кассовые чеки, остатки, цены конкурентов, логистика, колл-центр, мобильное приложение.


### 1. Стратегическая цель

Перейти от разрозненных витрин в реляционных БД и Excel к единой платформе данных: потоковая интеграция,
озеро данных, стандартизированные витрины и самообслуживание аналитики.


### 2. Текущее состояние

- Задержки отчетов 1-3 дня; несогласованные определения метрик;
- Пиковые нагрузки на хранилища при ночных операциях;
- Нет единого контура для событий.


### 3. Упрощенная целевая архитектура

- **Поток:** **Kafka** как шина событий продаж, остатков, цен; микросервисы публикуют контракты схем;
- **Пакет:** **S3-совместимое** хранилище + **Spark** для SQL по озеру;
- **Витрины:** слой **dbt** или аналог над **Spark SQL** / **warehouse**;
- **Потребители:** Business Intelligence, Data Scientist в Jupyter, операционные дашборды.


### 4. Этапы и горизонт

| Фаза | Срок (ориентир) | Результат |
|------|------------------|-----------|
| 0. Обследование | 1-2 мес. | Карта данных, приоритеты use-case, TCO |
| 1. Пилот | 3-4 мес. | Kafka-кластер (небольшой), 2-3 потока, озеро, одна витрина "продажи по дням/магазину" |
| 2. Масштабирование | 6-12 мес. | Каталог данных, CI/CD для пайплайнов, SLA, резервирование |
| 3. Зрелость | 12-24 мес. | Feature Store для ML, real-time аналитика, cost governance |


### 5. Ресурсы и бюджет

- **Команда**: архитектор данных, 2-3 инженера платформы, аналитик данных, представители бизнеса;
- **CapEx / OpEx**: лицензии (если облако), кластер **Kafka**, хранилище, вычисления **Spark**, мониторинг (**Prometheus** / **Grafana**), ИБ и аудит.


### 6. Риски и меры

- **Vendor lock-in** - приоритет открытых форматов (*Parquet*, *Iceberg*) и переносимых движков;
- **Качество данных** - контракты схем, мониторинг свежести и полноты потоков;
- **Безопасность** - шифрование, RBAC, маскирование ПДн, сегментация сети.


### 7. Критерии успеха

- Снижение времени подготовки ключевых отчетов;
- Доля запросов к самообслуживанию \> 50%;
- Задокументированные домены данных и ответственные владельцы.
